# Skill 4「一起动」测试框架

## 一、测试目标

验证 Skill 4 在以下维度的表现：
1. **路由准确性**：用户输入是否命中正确场景（A/B/C/D），是否正确拒绝不属于本 Skill 的请求
2. **输出完整性**：推荐结果是否包含 P0 必须要素（运动类型、时间、地点、人数、费用）
3. **调性一致性**：回复风格是否符合"朋友安利"而非"系统通知"
4. **规则遵守**：安全红线、数据诚实、行为边界是否被严格执行
5. **跨 Skill 引导**：是否在合适时机自然引导，不越界、不强推

### 3.4 sport-guide.md — 运动参考指南

**结构模板（每种运动一个条目）：**

```markdown
### 飞盘（Ultimate Frisbee）

- 参与门槛：零基础可参加，能跑能接就行
- 必须装备：运动鞋（平底防滑）、运动服
- 最好有：手套（防滑）、防晒
- 典型时长：1.5-2 小时
- 适合人数：8-14 人
```

## 四、数据采集与生成方案

### 4.1 各数据源的采集思路

| 数据表 | 采集方式 | 具体操作 |
|--------|---------|---------|
| activities.json | **半真实构造** | 参考小红书/飞盘社/Keep 约局帖子的真实格式，改写为模拟数据。不复制真实组织者信息。关注：标题怎么写、费用怎么描述、标签有哪些 |
| venues.json | **真实地点 + 模拟评分** | 场地名称和地址用真实地点（朝阳公园、奥森等公共场所），评分和特点字段为模拟数据 |
| weather.json | **人工构造** | 不需要采集，按测试场景需要手动编写。确保覆盖好天气/坏天气/极端天气 |
| sport-guide.md | **经验总结 + AI 辅助** | 可以让 AI 帮你生成初稿，再根据实际经验调整门槛描述和装备建议 |
| user-profiles.json | **虚构** | 完全虚构，服务于 Demo 展示 |

### 4.2 快速生成数据的步骤

**第 1 步：先写 weather.json（最简单，10 分钟）**
- 手动编写，7 天 × 3 城市
- 重点：确保至少有 2 天坏天气用于测试安全提醒

**第 2 步：再写 venues.json（30 分钟）**
- 打开地图查 15 个真实公共场地的名称和地址
- 手动补充 suitable_sports、features、rating

**第 3 步：然后写 activities.json（1 小时，最重要）**
- 先按分配清单建好 25 条数据的骨架
- 每条填入对应 venue 的地址信息（从 venues.json 引用）
- 标题和描述参考小红书约局帖的口语化风格
- 确保 date 字段和 weather.json 的日期对齐（同一天的活动要能被同一天的天气数据校验）

**第 4 步：写 sport-guide.md（30 分钟）**
- 10 种运动，每种 6 行，格式统一

**第 5 步：补充 user-profiles.json（10 分钟）**
- 在已有档案基础上加 4 个字段

### 4.3 数据质量检查清单

完成数据后，逐项核对：

- [ ] activities.json 中的 venue_name 都能在 venues.json 中找到对应
- [ ] activities.json 中的 date 都在 weather.json 的覆盖范围内
- [ ] activities.json 中 current_count ≤ capacity（不能报名人数超过上限）
- [ ] 至少有 2 条活动 current_count = capacity（已满），用于测试"已满"提示
- [ ] 至少有 3 条活动 current_count 接近 capacity（即将满），用于"还差 X 人"展示
- [ ] 每个预设用户的 city 都有对应的活动数据
- [ ] weather.json 中至少有 2 天 outdoor_friendly = false
- [ ] 所有 JSON 文件格式正确，能被解析
- [ ] activities 的 date 使用 Demo 展示日期的未来 7 天（需要在 Demo 前更新）
